Question:
    Create indexes so your service can quickly answer questions:
        - Which LEGO sets contain a specific brick type, regardless of color?
        - Which LEGO sets contain bricks of a specific color, regardless of type

Answer:
    Looking at the composite PK that I created in lego_inventory for task1 (set_id, brick_type_id, color_id),
    I will have to create two indexes in honor to have quick access to "LEGO sets based on a specific brick
    type" and "LEGO sets based on a specific color", one index on brick_type_id, and another on color_id. Thats
    because set_id is the first column in the composite PK, and the table will therefore be ordered first on
    set_id. Optionally, I could change the PK from test1 to: (brick_type_id, color_id, setId), that way I would
    only need to add one index on lego_inventory, specifically on color_id. I would not need an index on 
    brick_type_id because the table is already ordered based on this column from the PK. However, this would
    slow down set_id-first lookups.

Seing that the average time got worse after creating the index on color_id got me thinking. The reasonable explanation 
for this is that brick_type_id in my test is highly selective (meaning very few matches), so the index gives a big speedup. 
On the other hand, color_id = 1 is low-selective (meaning many matches), so using the index causes many heap lookups which
in this case is slower than a sequential scan (not using the index). The EXPLAIN ANALYZE results below supports this:

    With these results, I decided to keep the index on brick_type_id in lego_inventory, but deleted the other index on color_id
